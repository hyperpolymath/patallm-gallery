## Maintainer Responsibilities

Maintainers are responsible for:

1. **Code Review**
   - Review pull requests within 7 days
   - Provide constructive feedback
   - Ensure code quality and consistency
   - Verify RSR compliance

2. **Issue Triage**
   - Respond to issues within 48 hours
   - Label and prioritize issues appropriately
   - Close stale or resolved issues

3. **Release Management**
   - Follow semantic versioning
   - Maintain CHANGELOG.md
   - Tag releases appropriately
   - Announce releases to community

4. **Community Engagement**
   - Foster welcoming environment
   - Enforce Code of Conduct
   - Respond to questions and discussions
   - Mentor new contributors

5. **Documentation**
   - Keep documentation up-to-date
   - Review documentation PRs
   - Ensure accuracy of guides and examples

6. **Security**
   - Respond to security reports promptly
   - Coordinate security patches
   - Follow responsible disclosure practices

## Becoming a Maintainer

Contributors can become maintainers through demonstrated commitment:

1. **Criteria**
   - 10+ merged pull requests
   - 3+ months of consistent contribution
   - Demonstrated understanding of project goals
   - Positive community interactions
   - Alignment with project values

2. **Process**
   - Nomination by existing maintainer
   - Review by maintainer team
   - 2-week public comment period
   - Consensus approval (all maintainers agree)

3. **Onboarding**
   - Added to GitHub team with write access
   - Introduction to release process
   - Access to maintainer communication channels
   - Mentorship from existing maintainer

## Decision Making

### Consensus Model

- **Minor decisions**: Any maintainer can decide (documentation fixes, minor refactors)
- **Major decisions**: Consensus required (API changes, new features, policy changes)
- **Conflict resolution**: Discussion in GitHub Discussions, 7-day comment period, then vote

### Voting

- One vote per maintainer
- Simple majority for most decisions
- Unanimous for: removing maintainers, changing license, major architectural changes

## Stepping Down

Maintainers can step down gracefully:

1. Announce intention to step down
2. 2-week transition period to transfer knowledge
3. Move to Emeritus status
4. Recognition for contributions

## Inactive Maintainers

Maintainers inactive for 6+ months:

1. Maintainer team reaches out to check status
2. If no response in 2 weeks, privileges suspended
3. Can be reinstated upon return
4. Moved to Emeritus after 12 months inactive
